## 📋 执行流程

1. **第一阶段**：等待用户提供代码 → 分析代码 → 输出问题点
2. **第二阶段**：基于分析结果 → 设计新架构 → 等待确认
3. **第三阶段**：确认后 → 输出重构代码 → 完成交付

# 🧩 第一阶段：代码分析（必须先做）

我会提供代码，请你先完成：

1. 分析每个文件的作用
2. 梳理模块结构（目录 + 依赖关系）
3. 找出以下问题：

   * UI 布局问题（如乱用 absolute、结构混乱）
   * 状态管理问题（数据分散、重复定义）
   * 逻辑耦合问题（UI + 业务 + API 混在一起）
   * 可维护性问题（函数过长、复用性差）

📌 输出要求：

* 用"模块结构图 + 文字说明"描述
* 明确指出"必须重构的点"

⚠️ 此阶段不要写代码

# 🏗️ 第二阶段：重构设计（核心）

在分析基础上，设计新的架构：

## 1. 目录结构设计

分别给出：

ai-canvas/
map/

的完整新结构，例如：

* components/
* hooks/
* services/
* store/

## 2. 模块拆分方案

明确说明：

* 每个组件职责
* 每个 hook（useXXX）的职责

## 3. 状态管理方案

说明：

* 哪些状态是全局
* 哪些是局部

## 4. UI 改造方案

说明：

* 页面结构（顶部 / 内容 / 底部）
* 布局策略（flex 为主）
* 安卓适配注意点

📌 输出要求：

* 结构清晰
* 解释设计理由（为什么这样拆）

⚠️ 此阶段仍然不要写完整代码

# ⚙️ 第三阶段：代码重构（最后才做）

在前两阶段确认后，再输出代码：

## 要求：

1. 使用 Vue3 setup + composition API
2. 符合 uniapp x 规范
3. 代码必须模块化：

   * components
   * hooks（useCanvas / useMap / useAI / useLocation）
4. UI 和逻辑完全分离
5. 保留所有原有功能（不能删）

---

## 输出内容：

1. 新目录结构（完整）
2. 每个核心文件的代码（可运行）
3. 关键逻辑说明
4. 与旧代码对比的改进点

---

# ⚠️ 重要约束（必须遵守）

* 不允许删除已有功能
* 不允许把所有逻辑写在一个文件
* 不允许使用大量 inline style
* 不允许为了"看起来干净"而简化功能
* 必须保证安卓真机兼容

# 🧠 工作方式要求

* 每一阶段完成后等待我确认，再进入下一阶段
* 如果发现代码设计本身有严重问题，可以指出并给出替代方案
* 优先保证"可维护性 + 清晰结构"，而不是代码量少

# 🔧 Git 版本管理规范

1. 每次完成代码修改后，必须 commit
2. Commit 信息格式：`feat: 重构 xxx 模块` / `fix: 修复 xxx 问题`
3. 保持良好的版本历史，便于追溯
