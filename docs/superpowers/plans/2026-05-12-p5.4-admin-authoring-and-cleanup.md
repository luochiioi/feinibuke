# P5.4 Admin Authoring And UGC Cleanup Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use `superpowers:subagent-driven-development` (recommended) or `superpowers:executing-plans` to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Lock marker authoring to admin only, fix WeChat-mini-program login affordance, give themed routes an explicit reward kind, give the admin a proper task editor, and split task records out of the rewards page.

**Architecture:** Treat the App as a read-mostly consumer for markers/tasks/routes. All authoring lives in `uni-admin` + `admin-center`. The `rewards` collection stays the source of truth for points, gated by per-source `rewardKind`.

**Tech Stack:** uni-app x / UTS 5.07, uniCloud cloud objects, Node `node:test`, uni-admin Vue.

### Task 3: Route reward kind

**Problem:** The route create modal only has a free-text `reward` field. There is no way for the admin to mark a route as "points only" or "no reward at all"; meanwhile route completion always tries to write a rewards row, even for "no-reward" routes.

Reward-kind contract:

| `rewardKind` | UI label | `reward` text | Writes `rewards` row? | Points |
|---|---|---|---|---|
| `none` | 无奖励 | "" | No | 0 |
| `prize` | 奖品 | required | Yes, points 0 | from `parseRewardPoints` (likely 0) |
| `points` | 积分 | optional caption | Yes | from `rewardPoints` field |
| `both` | 奖品 + 积分 | required | Yes | from `rewardPoints` field |

- [ ] **Step 1: Write failing admin route-service tests**

Add tests:
- `validateRouteInput` rejects `rewardKind: 'prize'` without `reward` text.
- `validateRouteInput` accepts `rewardKind: 'none'` with empty reward and zero points.
- Legacy doc without `rewardKind` is normalized to `'prize'`.

Run: `node --test uniCloud-aliyun/cloudfunctions/admin-center/route-service.test.js` → fail.

- [ ] **Step 2: Implement validator + normalizer**

In `admin-center/route-service.js`: extend `validateRouteInput` and `buildRouteUpsertDoc` (or whatever exists) to set `rewardKind`, normalize `rewardPoints` to integer.

- [ ] **Step 3: Gate completion**

In `marker-center/route-completion.js`: `buildRouteRewardEntry` returns `null` when `rewardKind === 'none'`. The caller (`detectAndRecordCompletedRoutes` in `marker-center/index.obj.js`) must skip the `colRewards.add` step when the helper returns null. Add a `route-completion.test.js` case.

- [ ] **Step 4: Admin UI**

In `uni-admin/pages/routes/index.vue` modal: add a radio group "奖励类型 (无 / 奖品 / 积分 / 奖品+积分)". When kind is `points` or `both`, show an integer input for points. When kind is `none` or `points`, the `reward` text input is optional. Card meta shows kind.

- [ ] **Step 5: App route-detail**

In `pages/route-detail/route-detail.uvue`, the reward line renders based on kind: "无奖励" | reward text | "积分 N" | "{reward} · 积分 N".

- [ ] **Step 6: Verify**

Full test list, cloud syntax check, UTS grep.

Real-device acceptance:
1. Admin creates a "无奖励" route → user completes → no rewards row written, no celebration modal.
2. Admin creates a "积分" route with `rewardPoints: 50` → user completes → wallet pending +50; claim moves to issued.
3. Legacy routes still write a rewards row exactly as before (rewardKind defaulted to prize).
